    Presentation/Container

    Intro

    In the Presentation/Container Pattern, the naming and structure of files are crucial for maintaining clarity and scalability in your project. While there isn't a strict standard enforced across all React projects, there are widely accepted best practices and conventions for organizing your files and naming components.

    Here’s how you can structure your files and name your components in a way that follows the Presentation/Container Pattern:

    1. Folder Structure

    A common approach is to structure your components in a containers/ and components/ (or presentational/) folder. The containers will handle logic, while the presentational components will focus on UI rendering.

    src/
│
├── components/              # Presentational components
│   ├── Button.js            # Dumb component
│   ├── TodoItem.js          # Dumb component
│   └── TodoList.js          # Dumb component
│
├── containers/              # Container components (stateful)
│   ├── TodoContainer.js     # Smart component
│
└── App.js                   # App component, which might use containers

    In this structure:

    • components/: Contains dumb/presentational components that receive data via props and render it. These are reusable UI elements.
    • containers/: Contains smart/container components that manage state and handle business logic. These components typically pass props to the presentational components.

    2. Naming Conventions

    For Presentational (Dumb) Components:

    • Name them based on what they represent or how they display information.
    • Use descriptive, singular nouns or phrases.
    • These components should be UI-centric and stateless (or minimally stateful, if needed for UI reasons like toggles, form inputs, etc.).
    Example
    • TodoItem.js: Displays an individual todo.
    • Button.js: A simple button component.
    • Header.js: Renders a header UI.
    // components/TodoItem.js
import React from 'react';

const TodoItem = ({ todo }) => {
    return <li>{todo.text}</li>;
};

export default TodoItem;

    For Container (Smart) Components:

    • Name them based on their function or the data they manage.
    • The name should describe the container's responsibility (e.g., managing todos, user info).
    • It is common to name them with the suffix Container (e.g., TodoContainer.js or UserContainer.js).
    Example
    • TodoContainer.js: Fetches and manages the todo list, passing data to TodoItem.
    • UserContainer.js: Manages the user authentication and data, passing data to a presentational component like UserProfile.
    // containers/TodoContainer.js
import React, { useState, useEffect } from 'react';
import TodoItem from '../components/TodoItem';

const TodoContainer = () => {
    const [todos, setTodos] = useState([]);

    useEffect(() => {
        // Imagine fetching todos from an API
        const fetchTodos = async () => {
            const response = await fetch('/api/todos');
            const data = await response.json();
            setTodos(data);
        };

        fetchTodos();
    }, []);

    return (
        <ul>
            {todos.map(todo => (
                <TodoItem key={todo.id} todo={todo} />
            ))}
        </ul>
    );
};

export default TodoContainer;

    3. Use Clear and Descriptive File Names

    Be clear in naming so it’s immediately obvious which component is responsible for what:

    • Presentational Components: Prefer naming files after their role or function in the UI.

      • Button.js, Header.js, TodoItem.js

    • Container Components: Use the Container suffix to highlight that this component is responsible for logic and managing state.

      • TodoContainer.js, UserContainer.js, ProfileContainer.js

    This naming system allows you to easily understand the structure of the project just by looking at the file names.

    4. Organizing Components and Containers Together (Optional)

    In smaller projects, you might keep components and containers in the same directory or use subdirectories for a more modular approach.

    Example folder structure with modular subfolders 
    src/
│
├── features/                # Feature-based organization
│   ├── todos/
│   │   ├── components/      # Presentational components related to todos
│   │   │   ├── TodoItem.js
│   │   │   └── TodoList.js
│   │   └── containers/      # Containers for managing todos
│   │       └── TodoContainer.js
│   └── users/
│       ├── components/      # Presentational components related to users
│       └── containers/
│           └── UserContainer.js
│
└── App.js

    This approach groups related components by feature or domain rather than separating strictly by the component type (presentational vs. container).

    5. Component Folder with Index File

    In some cases, you can also use an index.js file inside each feature or container folder to simplify imports.

    Example 
    src/
│
├── components/
│   ├── TodoItem.js
│   └── index.js   # Exports all components in this folder
│
├── containers/
│   ├── TodoContainer.js
│   └── index.js   # Exports all containers in this folder
│
└── App.js

    index.js can look like this:

    // components/index.js
export { default as TodoItem } from "./TodoItem";

    This way, you can import components in TodoContainer.js like:

    import { TodoItem } from "../components";

    This keeps the import statements clean and scalable, especially as the project grows.

    Summary of Best Practices for File Structure and Naming:

    1. File Organization:

      • Store Presentational Components in a components/ folder.
      • Store Container Components in a containers/ folder.
      • Optionally, group by feature if your project scales.

    2. Naming Convention:

      • Presentational/Dumb Components: Name them based on their function or the UI they represent (e.g., TodoItem.js, Button.js).
      • Container/Smart Components: Use the Container suffix to indicate their role (e.g., TodoContainer.js, UserContainer.js).

    3. Modularity:

      • As the project grows, consider grouping components by features or domains (e.g., features/todos/, features/users/).

    4. Avoid Overcomplication:

      • Don’t over-structure—keep things simple in the beginning and scale as necessary.

    By following these practices, you'll keep your codebase organized, modular, and maintainable as it grows.

    Redux Testing

    Great! Lets dive deeper into testing Redux Thunk and Redux Saga to ensure that your asynchronous operations behave as expected in real-world applications.

    Testing Redux Thunk

    When testing actions that use Redux Thunk, we can use a mock store to dispatch actions and check that the correct actions are being dispatched based on the asynchronous logic.

    1. Testing Redux Thunk with Jest

      Scenario: You want to test the fetchProducts thunk action creator, which dispatches FETCH_PRODUCTS_REQUEST, FETCH_PRODUCTS_SUCCESS, or FETCH_PRODUCTS_FAILURE depending on the result of an API call.

      Test Setup:

      • Mocking the API: We'll mock the API call to return some fake data.
      • Mock Store: Use redux-mock-store to simulate Redux store behavior.

    Code:

    1. Install dependencies for testing:

      npm install redux-mock-store redux-thunk jest --save-dev

    2. Write the Test for Redux Thunk:

      import configureMockStore from 'redux-mock-store';
import thunk from 'redux-thunk';
import fetchMock from 'jest-fetch-mock'; // To mock fetch API
import { fetchProducts } from './actions'; // Thunk action
import { FETCH_PRODUCTS_REQUEST, FETCH_PRODUCTS_SUCCESS, FETCH_PRODUCTS_FAILURE } from './actionTypes';

const middlewares = [thunk];
const mockStore = configureMockStore(middlewares);

describe('fetchProducts action', () => {
    beforeEach(() => {
        fetchMock.resetMocks();
    });

    it('dispatches FETCH_PRODUCTS_SUCCESS after a successful API request', () => {
        const mockProducts = [{ id: 1, name: 'Product 1' }, { id: 2, name: 'Product 2' }];
        fetchMock.mockResponseOnce(JSON.stringify(mockProducts));

        const expectedActions = [
            { type: FETCH_PRODUCTS_REQUEST },
            { type: FETCH_PRODUCTS_SUCCESS, payload: mockProducts },
        ];

        const store = mockStore({ products: [] });

        return store.dispatch(fetchProducts()).then(() => {
            expect(store.getActions()).toEqual(expectedActions);
        });
    });

    it('dispatches FETCH_PRODUCTS_FAILURE when the API request fails', () => {
        const errorMessage = 'Failed to fetch';
        fetchMock.mockRejectOnce(new Error(errorMessage));

        const expectedActions = [
            { type: FETCH_PRODUCTS_REQUEST },
            { type: FETCH_PRODUCTS_FAILURE, payload: errorMessage },
        ];

        const store = mockStore({ products: [] });

        return store.dispatch(fetchProducts()).then(() => {
            expect(store.getActions()).toEqual(expectedActions);
        });
    });
});
      • Mock Store: redux-mock-store creates a mock version of the Redux store.
      • Action Sequence: The test ensures that the correct sequence of actions (FETCH_PRODUCTS_REQUEST, FETCH_PRODUCTS_SUCCESS, or FETCH_PRODUCTS_FAILURE) is dispatched based on the result of the API call.
      • Mock API Response: We use fetchMock to simulate different API responses (success and failure).

    Testing Redux Saga

    Testing Redux Saga involves verifying the flow of effects (call, put, take, etc.). We can use a library like redux-saga-test-plan to test sagas in isolation.

    1. Testing Redux Saga with Jest and redux-saga-test-plan

      Scenario: You want to test the fetchUserData saga, which fetches the user profile, notifications, and posts concurrently. You also want to ensure it retries the API calls up to 3 times if they fail.

    Code:

    1. Install the necessary testing libraries:

      npm install redux-saga-test-plan jest --save-dev

    2. Write the Test for Redux Saga:

      import { testSaga } from 'redux-saga-test-plan';
import { fetchUserData } from './sagas';
import { fetchProfileApi, fetchNotificationsApi, fetchPostsApi } from './api';
import { FETCH_PROFILE_SUCCESS, FETCH_NOTIFICATIONS_SUCCESS, FETCH_POSTS_SUCCESS } from './actionTypes';

describe('fetchUserData saga', () => {
    it('fetches profile, notifications, and posts concurrently', () => {
        const mockProfile = { name: 'John Doe' };
        const mockNotifications = [{ id: 1, message: 'New notification' }];
        const mockPosts = [{ id: 1, content: 'New post' }];

        testSaga(fetchUserData)
            .next()
            .all([
                call(fetchProfileApi),
                call(fetchNotificationsApi),
                call(fetchPostsApi),
            ])
            .next([mockProfile, mockNotifications, mockPosts])
            .put({ type: FETCH_PROFILE_SUCCESS, payload: mockProfile })
            .next()
            .put({ type: FETCH_NOTIFICATIONS_SUCCESS, payload: mockNotifications })
            .next()
            .put({ type: FETCH_POSTS_SUCCESS, payload: mockPosts })
            .next()
            .isDone();
    });

    it('retries API calls up to 3 times if they fail', () => {
        const error = new Error('API error');

        testSaga(fetchUserData)
            .next()
            .all([
                retry(3, 1000, fetchProfileApi),
                retry(3, 1000, fetchNotificationsApi),
                retry(3, 1000, fetchPostsApi),
            ])
            .throw(error)
            .put({ type: FETCH_PROFILE_FAILURE, payload: error.message })
            .next()
            .isDone();
    });
});
      • Test Flow: redux-saga-test-plan allows you to simulate the exact flow of your saga and assert each step (call, put, etc.).
      • Concurrent API Calls: We use all() to test that profile, notifications, and posts are fetched concurrently.
      • Retry Mechanism: The second test ensures that the API call retries up to 3 times using the retry() effect.

    Testing Best Practices for Redux Thunk and Redux Saga

    1. Mock External Dependencies:

      • For Thunk, mock API calls using jest-fetch-mock or similar libraries.
      • For Saga, use redux-saga-test-plan to simulate effects and test the sequence of actions.

    2. Test Action Sequences:

      • Ensure the correct actions are dispatched in the expected order.
      • For complex flows, check that the appropriate success or failure actions are dispatched based on the outcome.

    3. Isolate Logic:

      • Test action creators, reducers, and sagas separately to ensure that each piece of logic works independently before testing them together.

    Summary:

    • Thunk Testing: Use redux-mock-store to mock the Redux store and simulate dispatching asynchronous actions.
    • Saga Testing: Use redux-saga-test-plan to test the flow of saga effects like call, put, and all(), and simulate advanced logic such as retries.

    Testing Reducers

    Reducers are pure functions in Redux, meaning they should always produce the same output given the same input. This makes them easy to test. The goal of testing reducers is to ensure that they correctly update the state based on the dispatched actions.

    1. Basic Structure of a Reducer

    Lets say you have a reducer that handles fetching products:

```jsx
const initialState = {
    products: [],
    loading: false,
    error: null,
};

function productReducer(state = initialState, action) {
    switch (action.type) {
        case 'FETCH_PRODUCTS_REQUEST':
            return { ...state, loading: true };
        case 'FETCH_PRODUCTS_SUCCESS':
            return { ...state, loading: false, products: action.payload };
        case 'FETCH_PRODUCTS_FAILURE':
            return { ...state, loading: false, error: action.payload };
        default:
            return state;
    }
}
```

    Writing Unit Tests for Reducers

    Testing reducers is straightforward because they are pure functions. You only need to call the reducer with an action and an initial state, and then check if the output matches the expected state.

    2. Writing Tests for the Product Reducer

    Heres an example of how you would test this reducer using Jest:

    import productReducer from './productReducer';

describe('Product Reducer', () => {
  const initialState = {
    products: [],
    loading: false,
    error: null,
  };

  it('should return the initial state', () => {
    expect(productReducer(undefined, {})).toEqual(initialState);
  });

  it('should handle FETCH_PRODUCTS_REQUEST', () => {
    const action = { type: 'FETCH_PRODUCTS_REQUEST' };
    const expectedState = { ...initialState, loading: true };
    expect(productReducer(initialState, action)).toEqual(expectedState);
  });

  it('should handle FETCH_PRODUCTS_SUCCESS', () => {
    const action = {
      type: 'FETCH_PRODUCTS_SUCCESS',
      payload: [{ id: 1, name: 'Product 1' }],
    };
    const expectedState = {
      ...initialState,
      loading: false,
      products: action.payload,
    };
    expect(productReducer(initialState, action)).toEqual(expectedState);
  });

  it('should handle FETCH_PRODUCTS_FAILURE', () => {
    const action = { type: 'FETCH_PRODUCTS_FAILURE', payload: 'Error' };
    const expectedState = {
      ...initialState,
      loading: false,
      error: 'Error',
    };
    expect(productReducer(initialState, action)).toEqual(expectedState);
  });
});

    Best Practices for Testing Reducers

    1. Test for Default State:

      • Always test that the reducer returns the initial state when the state is undefined and the action is irrelevant (e.g., {}).

        it('should return the initial state', () => {
    expect(productReducer(undefined, {})).toEqual(initialState);
});

    2. Test Each Action Type:

      • Write a test case for each action type the reducer handles (FETCH_PRODUCTS_REQUEST, FETCH_PRODUCTS_SUCCESS, etc.). Ensure the state is updated as expected.

    3. Use Mocks for Action Payloads:

      • If an action payload is complex (e.g., an array of products), use mock data to ensure the test is focused on the reducer logic.

        const action = {
    type: 'FETCH_PRODUCTS_SUCCESS',
    payload: [{ id: 1, name: 'Mock Product' }],
};

    4. Test Edge Cases:

      • Consider testing edge cases like actions that are dispatched with unexpected payloads or malformed data.

    5. Avoid Testing Built-in Functions:

      • Dont test things like Object.assign or spread operators (...), as these are built-in JavaScript features. Focus on testing that your reducer logic is correct.

    Testing Redux with Thunk, Saga, and Reducers: Summary

    Now that youve seen how to test thunks, sagas, and reducers, heres a summary of the flow:

    1. Thunks:

      • Use redux-mock-store to mock the Redux store.
      • Mock external dependencies like APIs and verify that the correct actions are dispatched based on asynchronous logic.

    2. Sagas:

      • Use redux-saga-test-plan to simulate effects and check the flow of your saga (e.g., concurrent API calls, retry logic).
      • Focus on verifying the flow of effects (call, put, etc.).

    3. Reducers:

      • Test reducers by dispatching different actions and ensuring the state is updated correctly.
      • Write one test per action type, ensuring each actions impact on the state is validated.

    Testing React Components Connected to Redux

    When testing React components that are connected to Redux, we need to ensure:

    1. The component renders correctly with the expected state from Redux.
    2. The component dispatches the correct actions when interacting with UI elements (e.g., clicking buttons).

    We'll use react-testing-library (or @testing-library/react) and redux-mock-store to test connected components without relying on a real Redux store.

    1. Basic Structure of a Connected Component

    Lets assume we have a React component, ProductList, which is connected to Redux. It fetches products from the store and displays them, and it also has a button to reload the products.

    import React, { useEffect } from 'react';
import { connect } from 'react-redux';
import { fetchProducts } from './actions';

const ProductList = ({ products, loading, fetchProducts }) => {
  useEffect(() => {
    fetchProducts(); // Fetch products when component mounts
  }, [fetchProducts]);

  if (loading) {
    return <div>Loading...</div>;
  }

  return (
    <div>
      <h2>Product List</h2>
      <ul>
        {products.map((product) => (
          <li key={product.id}>{product.name}</li>
        ))}
      </ul>
      <button onClick={fetchProducts}>Reload Products</button>
    </div>
  );
};

const mapStateToProps = (state) => ({
  products: state.products.items,
  loading: state.products.loading,
});

export default connect(mapStateToProps, { fetchProducts })(ProductList);

    2. Writing Tests for the Connected Component

    Well test two scenarios:

    1. Rendering the component with initial state.
    2. Simulating a button click to dispatch the fetchProducts action.

    Setting Up Tests for Connected Components

    1. Install necessary libraries:

      npm install @testing-library/react redux-mock-store jest --save-dev

    2. Writing Tests with Mock Redux Store:

      We can use a mock Redux store to simulate different states in our tests. Heres how to set it up.

      import React from 'react';
import { render, screen, fireEvent } from '@testing-library/react';
import configureMockStore from 'redux-mock-store';
import { Provider } from 'react-redux';
import ProductList from './ProductList'; // The connected component
import thunk from 'redux-thunk';

// Mock store
const middlewares = [thunk];
const mockStore = configureMockStore(middlewares);

describe('ProductList Component', () => {
    let store;

    beforeEach(() => {
        // Initial mock state
        store = mockStore({
            products: {
                items: [{ id: 1, name: 'Product 1' }, { id: 2, name: 'Product 2' }],
                loading: false,
            },
        });
    });

    it('should render product list with products from Redux state', () => {
        render(
            <Provider store={store}>
                <ProductList />
            </Provider>
        );

        expect(screen.getByText('Product List')).toBeInTheDocument();
        expect(screen.getByText('Product 1')).toBeInTheDocument();
        expect(screen.getByText('Product 2')).toBeInTheDocument();
    });

    it('should dispatch fetchProducts action when "Reload Products" button is clicked', () => {
        render(
            <Provider store={store}>
                <ProductList />
            </Provider>
        );

        const reloadButton = screen.getByText('Reload Products');
        fireEvent.click(reloadButton);

        const actions = store.getActions();
        expect(actions).toEqual([{ type: 'FETCH_PRODUCTS_REQUEST' }]); // Adjust depending on the action dispatched
    });

    it('should show loading message when fetching products', () => {
        store = mockStore({
            products: { items: [], loading: true },
        });

        render(
            <Provider store={store}>
                <ProductList />
            </Provider>
        );

        expect(screen.getByText('Loading...')).toBeInTheDocument();
    });
});
      1. Mock Store: We create a mock Redux store with an initial state that mimics the real state in your app.
      2. Provider: Wrap the component in the Redux Provider and pass the mock store.

      3. Assertions:

        • Check if the component correctly renders data from the mock Redux state.
        • Simulate a button click and verify if the correct action is dispatched.

    Best Practices for Testing Connected Components:

    • Mock the Store: Use a mock store to simulate various Redux states (loading, error, etc.) to thoroughly test how your component handles different scenarios.
    • Test Action Dispatching: Simulate user interactions (like clicking a button) and verify if the correct Redux actions are dispatched.
    • Isolate UI Logic: Focus on testing the UIs interaction with Redux, not the internal logic of actions or reducers. These can be tested separately.

    Advanced Testing Scenarios for Redux-Connected Components

    1. Testing with Nested States:

      • If your Redux state has nested objects (e.g., state.products.items), ensure that you correctly mock and test the nested structure.

        // Example for a deeply nested state:

store = mockStore({
    products: {
        data: {
            items: [{ id: 1, name: 'Product 1' }],
        },
        loading: false,
    },
});

    2. Testing with Selectors:

      • If youre using selectors to derive data from the Redux store, you should mock the selectors output and test how the component renders based on it.

        import { getVisibleProducts } from './selectors';
jest.mock('./selectors');

it('should render visible products', () => {
    getVisibleProducts.mockReturnValue([{ id: 1, name: 'Visible Product' }]);

    render(
        <Provider store={store}>
        <ProductList />
        </Provider>
    );

    expect(screen.getByText('Visible Product')).toBeInTheDocument();
});

    3. Testing Middleware:

      • If youre using custom middleware or middleware like redux-thunk, ensure you test the interaction with middleware by checking the actions that are dispatched as a result of user interactions.

        const middlewares = [thunk]; // Include middleware in mock store setup

    4. Testing Complex Interactions:

      • For more complex interactions, simulate multiple steps (e.g., multiple button clicks or form submissions) and verify that the state and actions evolve correctly.

      Example: Testing a multi-step form that dispatches multiple actions:

      fireEvent.click(screen.getByText('Next Step'));
expect(store.getActions()).toEqual([
    { type: 'STEP_1_COMPLETE' },
    { type: 'LOAD_NEXT_STEP' },
]);

    Great! Lets dive into middleware testing and mocking API calls within Redux-connected components to handle asynchronous behavior.

    Testing Middleware in Redux

    Middleware in Redux is used to extend the functionality of the Redux store, allowing you to intercept actions before they reach the reducers. Common middleware like redux-thunk and redux-saga enable handling side effects such as API calls.

    1. Testing Thunk Middleware

    If you're using redux-thunk for async operations (like API calls), youll often want to ensure that:

    • Correct actions are dispatched before and after the async logic (e.g., LOADING, SUCCESS, ERROR).
    • The API calls (or any async logic) are correctly mocked.

    Example: Async Action with Thunk

    Lets assume you have a simple thunk that fetches products from an API:

    // actions.js
export const fetchProducts = () => {
  return async (dispatch) => {
    dispatch({ type: 'FETCH_PRODUCTS_REQUEST' });
    try {
      const response = await fetch('/api/products');
      const data = await response.json();
      dispatch({ type: 'FETCH_PRODUCTS_SUCCESS', payload: data });
    } catch (error) {
      dispatch({ type: 'FETCH_PRODUCTS_FAILURE', payload: error.message });
    }
  };
};

    Writing Tests for Thunks

    To test this, we mock the API call and check that the right actions are dispatched.

    import configureMockStore from 'redux-mock-store';
import thunk from 'redux-thunk';
import fetchMock from 'jest-fetch-mock'; // or axios-mock-adapter if you use axios
import { fetchProducts } from './actions';

// Setup mock store
const middlewares = [thunk];
const mockStore = configureMockStore(middlewares);

describe('fetchProducts thunk', () => {
  beforeEach(() => {
    fetchMock.resetMocks(); // Reset mock fetch calls
  });

  it('should dispatch FETCH_PRODUCTS_SUCCESS after successful API call', async () => {
    // Mock the API response
    fetchMock.mockResponseOnce(
      JSON.stringify([{ id: 1, name: 'Product 1' }])
    );

    const expectedActions = [
      { type: 'FETCH_PRODUCTS_REQUEST' },
      {
        type: 'FETCH_PRODUCTS_SUCCESS',
        payload: [{ id: 1, name: 'Product 1' }],
      },
    ];

    const store = mockStore({ products: [] });

    await store.dispatch(fetchProducts());

    expect(store.getActions()).toEqual(expectedActions);
  });

  it('should dispatch FETCH_PRODUCTS_FAILURE after a failed API call', async () => {
    // Mock a failed API response
    fetchMock.mockReject(new Error('Failed to fetch'));

    const expectedActions = [
      { type: 'FETCH_PRODUCTS_REQUEST' },
      { type: 'FETCH_PRODUCTS_FAILURE', payload: 'Failed to fetch' },
    ];

    const store = mockStore({ products: [] });

    await store.dispatch(fetchProducts());

    expect(store.getActions()).toEqual(expectedActions);
  });
});

    ??+ note "Key Points"

    1.  **Mocking API calls**:

    -   Use `fetch-mock` or `axios-mock-adapter` to simulate successful and failed API responses.

2.  **Asserting Dispatched Actions**:

    -   Verify that the correct actions (e.g., `REQUEST`, `SUCCESS`, `FAILURE`) are dispatched in the right order based on the API call.

    2. Testing Redux Saga Middleware

    If youre using redux-saga for side effects, testing involves simulating the side effects and asserting the correct sequence of events (e.g., dispatching actions, making API calls).

    Saga Example

    Lets say you have a saga that fetches products:

    // sagas.js
import { call, put, takeEvery } from 'redux-saga/effects';

function* fetchProductsSaga() {
  try {
    yield put({ type: 'FETCH_PRODUCTS_REQUEST' });
    const response = yield call(fetch, '/api/products');
    const data = yield call([response, 'json']);
    yield put({ type: 'FETCH_PRODUCTS_SUCCESS', payload: data });
  } catch (error) {
    yield put({ type: 'FETCH_PRODUCTS_FAILURE', payload: error.message });
  }
}

export function* watchFetchProducts() {
  yield takeEvery('FETCH_PRODUCTS', fetchProductsSaga);
}

    Testing Sagas with redux-saga-test-plan

    The redux-saga-test-plan library simplifies testing sagas by allowing you to simulate effects and assert their behavior.

    1. Install the library:

      npm install redux-saga-test-plan --save-dev

    2. Writing Tests for the Saga:

      Heres how you can test the fetchProductsSaga:

      import { runSaga } from 'redux-saga';
import { fetchProductsSaga } from './sagas';
import fetchMock from 'jest-fetch-mock';

describe('fetchProductsSaga', () => {
    beforeEach(() => {
        fetchMock.resetMocks(); // Reset mock fetch calls
    });

    it('should dispatch success action after fetching products', async () => {
        // Mock API response
        fetchMock.mockResponseOnce(JSON.stringify([{ id: 1, name: 'Product 1' }]));

        const dispatchedActions = [];

        // Run the saga
        await runSaga(
            {
                dispatch: (action) => dispatchedActions.push(action),
            },
            fetchProductsSaga
        ).toPromise();

        expect(dispatchedActions).toContainEqual({
            type: 'FETCH_PRODUCTS_SUCCESS',
            payload: [{ id: 1, name: 'Product 1' }],
        });
    });

    it('should dispatch failure action if API call fails', async () => {
        // Mock API error
        fetchMock.mockReject(new Error('API Error'));

        const dispatchedActions = [];

        // Run the saga
        await runSaga(
            {
                dispatch: (action) => dispatchedActions.push(action),
            },
            fetchProductsSaga
        ).toPromise();

        expect(dispatchedActions).toContainEqual({
            type: 'FETCH_PRODUCTS_FAILURE',
            payload: 'API Error',
        });
    });
});

      1. Use runSaga:

        • Run your saga with mocked dispatch and state functions to simulate side effects.

      2. Mock API Calls:

        • Mock the API calls (or any other side effects) to ensure that your saga correctly handles both success and failure cases.

      3. Verify Dispatched Actions:

        • After running the saga, check that the correct actions were dispatched (e.g., SUCCESS, FAILURE).

    Mocking API Calls in Redux-Connected Components

    Testing components that rely on async data (like API calls) means you need to mock those API responses and ensure the component behaves correctly.

    Example: ProductList Component with Thunk

    Lets take the ProductList component and test it with mocked API calls:

    import React, { useEffect } from 'react';
import { connect } from 'react-redux';
import { fetchProducts } from './actions';

const ProductList = ({ products, loading, fetchProducts }) => {
  useEffect(() => {
    fetchProducts();
  }, [fetchProducts]);

  if (loading) return <div>Loading...</div>;

  return (
    <div>
      <h2>Product List</h2>
      <ul>
        {products.map((product) => (
          <li key={product.id}>{product.name}</li>
        ))}
      </ul>
      <button onClick={fetchProducts}>Reload Products</button>
    </div>
  );
};

const mapStateToProps = (state) => ({
  products: state.products.items,
  loading: state.products.loading,
});

export default connect(mapStateToProps, { fetchProducts })(ProductList);

    Testing the Component with API Calls

    Well mock the API call and test the component rendering.

    import React from 'react';
import { render, screen, fireEvent } from '@testing-library/react';
import { Provider } from 'react-redux';
import configureMockStore from 'redux-mock-store';
import thunk from 'redux-thunk';
import fetchMock from 'jest-fetch-mock';
import ProductList from './ProductList';

// Mock store and thunk middleware
const middlewares = [thunk];
const mockStore = configureMockStore(middlewares);

describe('ProductList Component', () => {
  let store;

  beforeEach(() => {
    fetchMock.resetMocks(); // Reset the fetch mocks before each test

    store = mockStore({
      products: { items: [], loading: false },
    });
  });

  it('should fetch and display products', async () => {
    // Mock the API response
    fetchMock.mockResponseOnce(
      JSON.stringify([{ id: 1, name: 'Product 1' }])
    );

    render(
      <Provider store={store}>
        <ProductList />
      </Provider>
    );

    // Simulate component mounting and fetching products
    expect(fetchMock).toHaveBeenCalledTimes(1); // Ensure the API call was made

    // Assert that the product is rendered
    expect(await screen.findByText('Product 1')).toBeInTheDocument();
  });

  it('should display loading state', () => {
    store = mockStore({
      products: { items: [], loading: true },
    });

    render(
      <Provider store={store}>
        <ProductList />
      </Provider>
    );

    expect(screen.getByText('Loading...')).toBeInTheDocument();
  });
});

    Summary:

    • Thunk Testing: Simulate API calls and check dispatched actions using a mock store.
    • Saga Testing: Use runSaga to execute sagas and assert the side effects.
    • Component Testing: Mock API calls using fetchMock or similar tools and verify the component behavior.

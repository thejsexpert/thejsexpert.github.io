    JWT Client Side

    JWT on Client Side

    When handling JWT tokens on the client-side, the main focus should be usability rather than security, because the client doesn't have access to the secret or public keys needed to fully verify the signature of the JWT (which is the most critical part of the validation process). Instead, the client can perform certain checks that help improve the user experience and avoid relying on expired or malformed tokens.

    Here is the standard approach and what the client can verify:

    1. Verify JWT Structure (Format Check)

    The client can verify that the JWT is well-formed by checking if it has the correct structure. A valid JWT should consist of three parts (header, payload, and signature) separated by ..

    • Header: Encoded JSON containing metadata (e.g., algorithm).
    • Payload: Encoded JSON containing claims (e.g., user information, roles, and expiration).
    • Signature: Ensures the integrity and authenticity of the token (though this cannot be verified client-side).

    Why: Checking the structure ensures the token isn't malformed. If it doesn’t have three parts, it’s not a valid JWT.

    const isValidJWTStructure = (token) => {
  if (!token) return false;

  const parts = token.split(".");
  if (parts.length !== 3) return false;

  // Optionally, you could also check that each part matches base64 encoding
  const [header, payload, signature] = parts;
  return (
    /^[A-Za-z0-9-_]+$/.test(header) &&
    /^[A-Za-z0-9-_]+$/.test(payload) &&
    /^[A-Za-z0-9-_]+$/.test(signature)
  );
};

    2. Decode the JWT and Check Expiration

    One of the most common client-side checks is checking whether the token has expired. JWT tokens usually include an exp (expiration) claim in the payload, which you can decode and compare against the current time.

    Why: If the token is expired, it should not be used for further requests, and the client can prompt the user to re-authenticate.

    You can use libraries like jwt-decode to decode the token and read the expiration:

    import jwtDecode from "jwt-decode";

const verifyTokenExpiration = (token) => {
  try {
    const decoded = jwtDecode(token);
    const currentTime = Date.now() / 1000; // Current time in seconds

    // Check if the token has expired
    if (decoded.exp && decoded.exp < currentTime) {
      return false; // Token is expired
    }
    return true; // Token is valid
  } catch (error) {
    console.error("Failed to decode token", error);
    return false; // Token is invalid or malformed
  }
};

    3. Check the aud (Audience) and iss (Issuer) Claims (Optional)

    You can optionally check the aud (audience) and iss (issuer) claims to ensure the token was issued for the correct application (audience) and by a trusted issuer. This isn't typically done on the client side in many apps, but for additional security checks (especially for more complex applications), you can verify that these fields are correct.

    • aud (Audience): This should be your application’s identifier or a list of valid audiences.
    • iss (Issuer): This should be the issuer of the token (e.g., your identity provider or backend server).

    Why: This ensures the token was issued for the intended audience and by a trusted party.

    const verifyAudienceAndIssuer = (token) => {
  try {
    const decoded = jwtDecode(token);

    const expectedAudience = "myAppAudience";
    const expectedIssuer = "myIssuer";

    if (decoded.aud !== expectedAudience || decoded.iss !== expectedIssuer) {
      return false; // Token has an unexpected audience or issuer
    }
    return true; // Token is valid for this audience and issuer
  } catch (error) {
    return false; // Token is invalid or malformed
  }
};

    4. Handle Token Expiration Gracefully

    While the client can't fully validate the signature, it can check for expiration (as shown above) and manage the user session accordingly. You should implement a flow where expired tokens are detected and users are logged out or prompted to re-authenticate.

    • If the token has expired, clear any user session information (like storing it in localStorage or sessionStorage) and redirect the user to the login page.
    • Optionally, you can implement a refresh token flow (with a separate API endpoint) to silently refresh the session without requiring the user to log in again.

    5. Gracefully Handle Invalid Tokens

    If the JWT is malformed or invalid, it’s important to catch errors gracefully so the app doesn’t crash or behave unexpectedly. This is especially important if you are decoding the token on the client side.

    For instance, in the code snippet above where jwtDecode is used, you might want to ensure that any invalid token (one that can't be decoded) is ignored and the user is redirected to the login page.

    try {
  const decodedToken = jwtDecode(token);
  if (decodedToken.exp < Date.now() / 1000) {
    // Handle expired token
    alert("Session expired, please log in again.");
  }
} catch (error) {
  alert("Invalid token.");
  // Clear the session and redirect to login page
  localStorage.removeItem("token");
  window.location.href = "/login";
}

    6. Store Tokens Securely

    While not a "validation" per se, it’s essential to store JWT tokens securely on the client side. Here are some security tips:

    • Use HttpOnly cookies: This is the safest way to store JWT tokens, as it prevents JavaScript from accessing the token (mitigates XSS attacks).
    • Avoid localStorage or sessionStorage for storing sensitive tokens because these are accessible via JavaScript and could be stolen via XSS attacks. If you must store them in localStorage, ensure you're using proper XSS defenses.

    Standard Practice Overview:

    • Verify JWT structure: Ensure the token has three parts (header, payload, signature).
    • Decode and check expiration: Verify that the token has not expired (check the exp claim).
    • Optionally verify aud and iss claims: Check if the token's audience and issuer are valid for your application.
    • Gracefully handle invalid or expired tokens: Log out users or prompt them to log in again when the token is expired or malformed.
    • Secure token storage: Use HttpOnly cookies or other secure methods to store JWT tokens.

    What Shouldn't the Client Do?

    • Never verify the JWT signature: The client doesn't have the secret or public key required to verify the integrity of the JWT, so you should rely on the server for actual signature verification.
    • Don’t trust the JWT entirely on the client-side: Always assume that the client-side data can be tampered with. For example, if you rely on decoding a JWT to determine the user's role, make sure that your server performs its own authorization checks when handling sensitive requests.

    Conclusion:

    In summary, the client-side checks you should perform for a JWT are:

    1. Verify the token structure.
    2. Decode and check expiration (if the token is expired, log the user out).
    3. Optionally verify claims like aud and iss.
    4. Ensure the token is securely stored.

    The server-side is where the real trust and security come from—so always ensure that the server verifies the JWT's signature and performs authorization checks.

    Summary of Compulsory vs. Optional Client-Side Checks

    When it comes to JWT verification on the client-side, there are some checks that are compulsory to ensure the token is at least usable and does not immediately violate basic expectations (like being expired or malformed), while others are optional, depending on your application's specific security and user experience requirements.

    Compulsory Checks (Must Do on the Client-Side)

    These are the checks that should always be done when handling JWT tokens on the client side to ensure the token is safe to use and doesn't immediately cause issues in the application.

    1. Verify JWT Structure (Well-formed Token)

    A valid JWT token consists of three parts: the header, the payload, and the signature, separated by dots (.). You should always check the token format to ensure it's properly structured.

    • Why: Ensures the token is a valid JWT format and isn't malformed.
    • What to do: Check that the token is split into three parts and that each part is a valid Base64 URL string.
    const isValidJWTStructure = (token) => {
    if (!token) return false;
    const parts = token.split('.');
    if (parts.length !== 3) return false;
    return /^[A-Za-z0-9-_]+$/.test(parts[0]) && /^[A-Za-z0-9-_]+$/.test(parts[1]) && /^[A-Za-z09-_]+$/.test(parts[2]);
};

    2. Decode and Check Expiration (exp Claim)

    The expiration (exp) claim is a standard claim in JWT that indicates the token's expiry time. If the token has expired, it should not be used, and the user should be logged out or prompted to re-authenticate.

    • Why: Ensures the token is still valid and hasn’t expired, preventing issues with session management.
    • What to do: Decode the JWT to check the exp claim and compare it to the current time.
    import jwtDecode from 'jwt-decode';

const verifyTokenExpiration = (token) => {
    try {
        const decoded = jwtDecode(token);
        const currentTime = Date.now() / 1000; // Current time in seconds

        // Check if the token has expired
        if (decoded.exp && decoded.exp < currentTime) {
            return false; // Token expired
        }

        return true; // Token still valid
    } catch (error) {
        return false; // Malformed token
    }
};
    Optional Checks (Useful, But Not Required)

    These checks are not mandatory but can enhance your security posture and improve user experience, depending on your application's specific needs.

    3. Verify Audience (aud) Claim

    The audience (aud) claim indicates for which system the token is intended. You can verify this claim to ensure that the token was issued for your specific application.

    • Why: To prevent token misuse where the token might be intended for another audience.
    • What to do: Decode the token and check if the aud claim matches your application's expected value.
    const verifyAudience = (token) => {
    try {
        const decoded = jwtDecode(token);
        const expectedAudience = 'yourAppIdentifier'; // Example audience value
        return decoded.aud === expectedAudience;
    } catch (error) {
        return false;
    }
};

    4. Verify Issuer (iss) Claim

    The issuer (iss) claim identifies who issued the token (usually your authentication service). You can check this claim to ensure the token comes from a trusted source.

    • Why: To ensure the token is from a trusted issuer.
    • What to do: Decode the token and verify the iss claim.
    const verifyIssuer = (token) => {
    try {
        const decoded = jwtDecode(token);
        const expectedIssuer = 'yourTrustedIssuer'; // Example issuer value
        return decoded.iss === expectedIssuer;
    } catch (error) {
        return false;
    }
};

    5. Verify Custom Claims

    Some applications might include custom claims in the JWT payload, like role, permissions, or specific user attributes (e.g., is_admin). You can decode the JWT and check if these claims are valid and match what the client needs.

    • Why: To tailor the app's behavior based on the user's roles or permissions.
    • What to do: Decode the token and check for custom claims like role, permissions, etc.
    const verifyRoleClaim = (token) => {
    try {
        const decoded = jwtDecode(token);
        const requiredRole = 'admin'; // Example role
        return decoded.role === requiredRole;
    } catch (error) {
        return false;
    }
};
    What Shouldn't the Client Do? (What is Not Safe or Recommended)

    1. Never Verify the JWT Signature

    The client cannot verify the signature of a JWT because it does not have access to the secret key or the public key used to sign the token. Signature verification is the server’s responsibility. If you were to attempt signature verification on the client, you'd expose the secret key (which defeats the purpose of using JWTs).

    • Why: The signature verification requires the secret or public key, which should never be exposed to the client.

    • What to do: Always rely on the server to handle JWT signature verification.

    2. Never Trust the JWT Claims Without Server Validation

    The client-side validation can help manage the user interface and session flow, but it should never be the sole decision-maker on whether a user is authorized or not. The server should always perform proper authorization checks for sensitive actions.

    • Why: The client could be compromised (e.g., via XSS), so it cannot be fully trusted for critical security decisions.

    • What to do: Always ensure that authorization and access control checks happen server-side.

    Compulsory Client-Side Checks (Must Do):

    1. Verify JWT structure (format): Ensure the token has 3 parts.
    2. Decode and check expiration (exp claim): Ensure the token has not expired.

    Optional Client-Side Checks (Helpful but Not Required):

    1. Verify aud (Audience) claim: Ensure the token is intended for your app.
    2. Verify iss (Issuer) claim: Ensure the token is issued by a trusted issuer.
    3. Verify custom claims (e.g., role, permissions): Check custom data (if needed) for UI behavior.

    What the Client Should Never Do:

    1. Never verify the JWT signature (this should be done server-side).
    2. Never trust client-side JWT claims for sensitive operations (authorization should be enforced server-side).
    Conclusion
    • Compulsory: The client should always verify the token structure and check for expiration. This ensures that the token isn't malformed and hasn't expired, which could cause usability issues or security risks.
    • Optional: Verifying the aud (audience) and iss (issuer) claims, as well as custom claims like role, can be useful for certain applications but are not strictly necessary for basic functionality.
    • Never: The client should never verify the JWT signature or rely on the client-side for authorization decisions. Always trust the server to do proper validation.

    By following these guidelines, you can manage JWT tokens safely on the client-side while ensuring that security-sensitive actions (like signature verification and authorization) are handled on the server-side where they belong.

    USAGE

    Steps to Update Your Code:
    1. Check the JWT Structure: We'll check if the JWT has 3 parts (header, payload, signature).
    2. Check Expiration (exp Claim): We'll decode the token and check if it has expired.
    3. If Token is Valid: Store the token in Redux and localStorage.
    4. If Token is Invalid/Expired: Show an error notification.
    import React, { useState } from "react";
import { useNavigate } from "react-router-dom";
import { Layout, Spin, Form, Input, Button, Alert, notification } from "antd";
import { useDispatch } from "react-redux";
import queryString from "query-string";
import jwtDecode from "jwt-decode"; // Correct import for jwt-decode
import { useLoginMutation } from "@features/auth/services/authApi";
import { setAuth } from "@features/auth/services/authSlice";
import { LoginHeader, LoginForm, LoginFooter } from "./";
import { notify } from "@shared/utils/notificationUtils";
import logger from "@shared/utils/logger";

const { Content } = Layout;

const Login = () => {
const [login, { isLoading, isError, data, error }] = useLoginMutation();
const [loading, setLoading] = useState(false);
const dispatch = useDispatch();
const navigate = useNavigate();

// Helper function to check token expiration
const isTokenExpired = (token) => {
    try {
        const decoded = jwtDecode(token);
        const currentTime = Date.now() / 1000; // Current time in seconds
        return decoded.exp && decoded.exp < currentTime;
    } catch (error) {
        return true; // In case the token is malformed
    }
};

// Helper function to verify token structure (split into 3 parts)
const isValidJWTStructure = (token) => {
    const parts = token.split(".");
    return (
        parts.length === 3 && parts.every((part) => /^[A-Za-z0-9-_]+$/.test(part))
    );
};

const onFinish = async (values) => {
    setLoading(true);
    const { username, password } = values;

    try {
        // Prepare the form data to send
        const normalData = { username, password, expiresInMins: 1 };
        const user = await login(normalData).unwrap(); // Call the login mutation

        const accessToken = user?.accessToken;

        // 1. Validate the JWT Structure
        if (!isValidJWTStructure(accessToken)) {
            notify(
                "error",
                "topRight",
                "Invalid Token",
                "The token structure is invalid.",
                2
            );
            setLoading(false);
            return;
        }

        // 2. Check if the token is expired
        if (isTokenExpired(accessToken)) {
            notify(
                "error",
                "topRight",
                "Token Expired",
                "The token has expired. Please login again.",
                2
            );
            setLoading(false);
            return;
        }

        // 3. Decode the token
        const decodedToken = jwtDecode(accessToken);
        logger.info({ msg: "Decoded Token", decodedToken });

        // 4. Dispatch the token and user info to Redux store
        dispatch(
            setAuth({
                token: accessToken || null,
                refresh: user?.refreshToken || null,
                role: decodedToken?.role || null,
                scope: decodedToken?.scope || null,
            })
        );

        // 5. Store the token in localStorage for persistence
        localStorage.setItem("authToken", accessToken);

        // 6. Redirect the user to the dashboard
        navigate("/dashboard");
        setLoading(false);
    } catch (err) {
        setLoading(false);
        logger.error({ msg: "Login Error", err });
        notify(
            "error",
            "topRight",
            err?.data?.error || "Invalid Credentials",
            err?.data?.error_description ||
            "An unexpected error occurred during login.",
            2
        );
    }
};

    return (
        <Layout style={{ minHeight: "100vh", width: "100%" }}>
            <LoginHeader />

            <Content
                style={{
                    padding: "50px 0",
                    display: "flex",
                    justifyContent: "center",
                    width: "100%",
                }}
            >
                <div
                style={{
                    width: "100%",
                    maxWidth: 420,
                    padding: "20px",
                }}
                >
                    <LoginForm onFinish={onFinish} loading={loading} />
                </div>
            </Content>

            <LoginFooter />
        </Layout>
    );
};

export default Login;

    Utility Functions

    Creating the Utility Functions
    1. validateJWTStructure: Checks if the JWT has the correct structure (i.e., 3 parts: header, payload, signature).
    2. isTokenExpired: Checks if the JWT is expired by comparing the exp claim to the current timestamp.
    3. decodeToken: Decodes the JWT (if needed for extracting additional information such as role, scope, etc.).

    Let's implement these utility functions in a dedicated file, such as jwtUtils.js.

    Step 1: Create Utility Functions

    Create a new file, e.g., src/shared/utils/jwtUtils.js.

    // src/shared/utils/jwtUtils.js

// WRONG
// import jwtDecode from "jwt-decode";
// CORRECT
import { jwtDecode } from "jwt-decode";

/**
 * Check if the JWT has the correct structure.
 * A valid JWT should consist of 3 parts: header, payload, signature.
 * @param {string} token - The JWT string.
 * @returns {boolean} - True if the structure is valid, otherwise false.
 */
export const validateJWTStructure = (token) => {
    const parts = token.split(".");
    return (
        parts.length === 3 && parts.every((part) => /^[A-Za-z0-9-_]+$/.test(part))
    );
};

/**
 * Check if the JWT is expired.
 * @param {string} token - The JWT string.
 * @returns {boolean} - True if the token is expired, otherwise false.
 */
export const isTokenExpired = (token) => {
    try {
        const decoded = jwtDecode(token);
        const currentTime = Date.now() / 1000; // Current time in seconds
        return decoded.exp && decoded.exp < currentTime;
    } catch (error) {
        return true; // Return true if the token is malformed or invalid
    }
};

/**
 * Decode the JWT and return its decoded payload.
 * @param {string} token - The JWT string.
 * @returns {object|null} - The decoded payload or null if invalid.
 */
export const decodeToken = (token) => {
    try {
        return jwtDecode(token);
    } catch (error) {
        return null; // Return null if the token is invalid or malformed
    }
};

    Refresh Token Verification

    Yes, it is a good practice to verify the refresh token's structure as well, even though the refresh token itself is typically simpler than the access token. Verifying the structure of the refresh token ensures that it conforms to the expected JWT format before you attempt to use it for a refresh operation.

    Why Verify the Refresh Token Structure?

    • Security: Just like the access token, the refresh token can be tampered with or malformed. Ensuring that the refresh token follows the correct structure prevents attempts to use invalid or corrupted tokens.
    • Prevention of Errors: Verifying the structure helps you avoid errors in the flow before you make the request to your backend to refresh the token. If the structure is incorrect, you can notify the user early without attempting to use the token in an API call.

    When to Verify the Refresh Token?

    • Before sending the refresh token to the server: If the token is invalid or malformed, you would want to stop the process immediately and avoid making an unnecessary request to your backend.
    • In your frontend application: In some cases, you may want to validate the structure of the refresh token stored in localStorage or passed in requests to ensure that it is in the correct JWT format.

    Step 2: Using the validateJWTStructure in the Refresh Token Flow

    Now let's update the refreshToken function to verify the structure of the refresh token before attempting the refresh process:

    // src/shared/utils/jwtUtils.js

import axios from "axios";
import { notify } from "@shared/utils/notificationUtils";

// The function to refresh the token
export const refreshToken = async (refreshToken) => {
    try {
        // First, check if the refresh token structure is valid
        if (!validateJWTStructure(refreshToken)) {
            notify("error", "topRight", "Invalid Refresh Token", "The refresh token structure is invalid.", 2);
            throw new Error("Invalid refresh token structure");
        }

        // Make an API call to your backend to refresh the token
        const response = await axios.post("/api/auth/refresh-token", {
            refresh_token: refreshToken,
        });

        // Check if the response contains a new access token
        if (response.data?.accessToken) {
            return response.data.accessToken; // Return the new access token
        }

        throw new Error("Failed to refresh the token.");
    } catch (error) {
        console.error("Refresh token error", error);
        notify("error", "topRight", "Session Expired", "Please log in again.", 3);
        throw error; // Rethrow the error to be handled downstream
    }
};

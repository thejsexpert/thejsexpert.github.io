    How to Use JavaScript Design Patterns in Daily Applications (Simple Examples)

    Heres how you can use the most common design patterns in JavaScript in your daily applications:

    1. Module Pattern

    The Module Pattern helps organize your code by encapsulating related functions into a single object, making your code more modular and reusable. Its great for managing separate functionality like API services or UI components.

    // UserService.js (Module)
const UserService = (function () {
    let users = [];

    function addUser(user) {
        users.push(user);
        console.log("User added:", user);
    }

    function getUsers() {
        return users;
    }

    return {
        addUser,
        getUsers,
    };
})();

// Usage
UserService.addUser({ name: "John", age: 30 });
UserService.addUser({ name: "Jane", age: 25 });
console.log(UserService.getUsers()); // [{ name: 'John', age: 30 }, { name: 'Jane', age: 25 }]

    When to Use:

    • Whenever you need to manage different pieces of functionality in your app separately (e.g., user management, data fetching).

    2. Observer Pattern

    The Observer Pattern is useful when you have multiple parts of your app that need to react to changes, like a UI updating in response to data changes.

    // Subject.js
class Subject {
    constructor() {
        this.observers = [];
    }

    subscribe(observer) {
        this.observers.push(observer);
    }

    notify(data) {
        this.observers.forEach(observer => observer.update(data));
    }
}

// Observer.js
class Observer {
    update(data) {
        console.log('Received data:', data);
    }
}

// Usage
const subject = new Subject();
const observer1 = new Observer();
const observer2 = new Observer();

subject.subscribe(observer1);
subject.subscribe(observer2);

subject.notify('New Data Available');
// Both observers will log: "Received data: New Data Available"

    When to Use:

    • Use it when you need to notify multiple components or modules when a certain event occurs (e.g., real-time notifications or data updates).

    3. Factory Pattern

    The Factory Pattern is helpful when you need to create different types of objects based on certain conditions, like different kinds of UI components or products.

    // ButtonFactory.js
function ButtonFactory(type) {
    if (type === 'primary') {
        return { text: 'Primary Button', style: 'blue' };
    } else if (type === 'secondary') {
        return { text: 'Secondary Button', style: 'gray' };
    } else {
        return { text: 'Default Button', style: 'default' };
    }
}

// Usage
const primaryButton = ButtonFactory('primary');
const secondaryButton = ButtonFactory('secondary');

console.log(primaryButton); // { text: 'Primary Button', style: 'blue' }
console.log(secondaryButton); // { text: 'Secondary Button', style: 'gray' }

    When to Use:

    • When you need to create objects based on some conditions, such as dynamically generating UI elements (e.g., buttons, forms).

    4. Singleton Pattern

    The Singleton Pattern is useful when you need a single instance of an object or service across your entire application, like a logger or configuration manager.

    // Logger.js (Singleton)
const Logger = (function () {
    let instance;

    function createInstance() {
        return {
        log: function (message) {
            console.log("Log message:", message);
        },
        };
    }

    return {
        getInstance: function () {
        if (!instance) {
            instance = createInstance();
        }
        return instance;
        },
    };
})();

// Usage
const logger1 = Logger.getInstance();
const logger2 = Logger.getInstance();

logger1.log("This is the first log"); // "Log message: This is the first log"
logger2.log("This is the second log"); // "Log message: This is the second log"

console.log(logger1 === logger2); // true (same instance)

    When to Use:

    • Use it when you need a shared instance of an object (e.g., a logging service or configuration settings) throughout your app.

    5. Decorator Pattern

    The Decorator Pattern allows you to add extra functionality to an existing object without modifying its structure, like adding logging or validation.

    // Original function
function greet(name) {
    return `Hello, ${name}`;
}

// Decorator to log function calls
function logDecorator(fn) {
    return function (...args) {
        console.log(`Calling function with args: ${args}`);
        return fn(...args);
    };
}

// Usage
const decoratedGreet = logDecorator(greet);
console.log(decoratedGreet("John")); // "Calling function with args: John" followed by "Hello, John"

    // logDecorator.js
function logDecorator(fn) {
    return function(...args) {
        console.log(`Calling function with args: ${args}`);
        return fn(...args);
    };
}

export default logDecorator;

    // greet.js
function greet(name) {
    return `Hello, ${name}`;
}

export default greet;

    // main.js
import logDecorator from './logDecorator.js';
import greet from './greet.js';

const decoratedGreet = logDecorator(greet);
console.log(decoratedGreet('John'));  // "Calling function with args: John" followed by "Hello, John"

    When to Use:

    • When you need to add extra functionality (e.g., logging, validation, caching) to existing functions or methods without altering their core logic.
    File Explained

    This folder structure with file names makes it easier to integrate design patterns into your daily JavaScript workflow. Here's a summary of file names and what they do:

    1. Module Pattern: Encapsulate logic in modules (UserService.js).
    2. Observer Pattern: Build an event-driven system with subscribers (Subject.js, Observer.js).
    3. Factory Pattern: Dynamically create objects based on input (ButtonFactory.js).
    4. Singleton Pattern: Ensure only one instance of a service or resource (Logger.js).
    5. Decorator Pattern: Add functionality to functions without changing them (logDecorator.js, greet.js).

    By splitting your project into smaller, pattern-based files, you'll achieve better maintainability and code organization in large-scale applications.

    Conclusion:

    Heres how you can use these design patterns in daily JavaScript applications:

    1. Module Pattern: Organize your code into reusable modules, great for services like user management or API handling.
    2. Observer Pattern: Use when you need event-driven behavior, like updating multiple parts of the UI when data changes.
    3. Factory Pattern: Dynamically create objects based on conditions, perfect for generating UI components.
    4. Singleton Pattern: Ensure only one instance of a service or configuration manager is created, such as a logger.
    5. Decorator Pattern: Add extra functionality to existing objects (e.g., logging, validation) without modifying their structure.

    These patterns help make your code more structured, maintainable, and scalable as your JavaScript applications grow.
